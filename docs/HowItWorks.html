🤖 Multi-Agent Dev Studio

How the orchestration pipeline works — from feature request to polished implementation

For Users
For Developers

What It Does

You type a feature request — like "Add a retry decorator with exponential backoff" — and three Claude AI agents collaborate to produce a complete, reviewed Python implementation through a sequential pipeline. The spec is written and quality-checked before code generation begins. You get a quality-gated feature spec, the code, and a trace of the review process, all in one output.

Step by Step

1

You submit a feature request

Type your request in the input box (up to 2000 characters) and click Run Pipeline. The pipeline validates your input instantly — empty or too-long requests are rejected before any API calls are made.

2

Spec Agent writes the spec

The Spec Agent (Claude Haiku) writes a feature spec including a user-value overview, GIVEN/WHEN/THEN acceptance criteria, design decisions, and out-of-scope items. This runs before any code generation.

3

Spec review gate checks quality

An Orchestrator (Claude Sonnet) evaluates the spec against 5 strict criteria: user value articulated, at least 3 GIVEN/WHEN/THEN scenarios including an error case, at least 2 design decisions covering data structures and error handling, plausible out-of-scope exclusions, and internal consistency. If the spec falls short and retries remain, the Spec Agent is asked to revise. Only when the spec passes does code generation begin.

4

Code Agent generates the implementation

The Code Agent (Claude Sonnet) writes a typed Python implementation using the validated spec. On fix cycles, it opens with a ## Issues Addressed section listing exactly which numbered issues from the reviewer it has fixed.

5

A reviewer checks alignment and fix claims

The Review Agent (Claude Sonnet) checks 8 criteria: spec completeness, code↔spec alignment, type safety, edge cases, no hallucinated imports, DRY, single responsibility, and error handling. Issues are ranked [P1] (critical), [P2] (important), [P3] (polish). On subsequent cycles, the reviewer also verifies each claimed fix from the code agent's ## Issues Addressed section.

6

Fix cycles run automatically (if needed)

If the reviewer flagged issues, the Code Agent re-runs with the numbered issue list. The Code Agent must acknowledge each fix by number. This continues until the code passes review or the iteration cap is reached (default: 1 cycle).

7

Final output delivered

Once approved, a Synthesis Agent (Claude Sonnet) combines everything into a structured markdown document: the feature spec, the implementation in a code block, a review trace showing how many cycles it took, and — if the spec had unresolved gaps — a spec gap notes section.

Tips

  • Be specific — "Add a retry decorator with exponential backoff and max retries" produces better output than "add retries". More specific requests also pass the spec gate on the first attempt.
  • Focused scope — the pipeline is designed for self-contained Python modules or HTML/JS components. Full applications ("build a CRM") will be rejected by the scope guardrail.
  • Max iterations — if you see the warning banner, the output is still useful — just not fully reviewed. Submit again with a more focused request.
  • Spec gap notes — if the spec review budget is exhausted without full approval, the final output will include a spec gap notes section describing what remained unresolved.
  • Example prompts — click any example below the input box to pre-fill it.

Key Design Decisions

DecisionChoiceRationale
Execution orderSequential pipeline (spec → spec_review gate → code → review)Spec must be validated before code generation. Parallel fan-out produced lower-quality code because code_agent had no reviewed spec to work from.
Model tiersSonnet orchestrator (spec_review + review + synthesis), Haiku spec agent, Sonnet code agentSpec writing requires less judgment (Haiku sufficient). Code generation requires stronger synthesis (Sonnet needed — Haiku produced truncated output and undefined function errors). Review/judge tasks use Sonnet — same model as code but highest context utilization.
Review feedbackNumbered [P1]/[P2]/[P3] issues + acknowledgement trackingCode agent references issues by number in ## Issues Addressed; reviewer verifies each claim in the next cycle. Prevents silent non-fixes.
State schemaCustom AgentState TypedDictTask-output-centric; maps cleanly to the domain vs. message-centric MessagesState
UI frameworkGradio BlocksLowest friction for Python-native demo; streaming generators for live updates; no separate frontend process
Secret handlingPydantic SecretStrAPI key never appears in logs, repr, or model serialization

Graph State

class AgentState(TypedDict):
    feature_request:         str                            # user input
    spec_output:             SubAgentOutput | None          # latest spec
    code_output:             SubAgentOutput | None          # latest code
    review_feedback:         ReviewFeedback | None          # latest review
    iteration_count:         int                            # increments each review cycle
    spec_review_iteration:   int                            # increments each spec gate retry
    spec_gap_notes:          str                            # unresolved spec gaps (if gate exhausted)
    code_fix_acknowledgement: str                           # ## Issues Addressed section from code_agent
    final_output:            str | None                     # set by synthesize
    status:                  Literal["running",
                                     "approved",
                                     "max_iterations_reached"]
    review_history:          Annotated[list[ReviewFeedback],
                                       operator.add]        # append-only across cycles

Each node returns only the keys it produces — e.g. {"code_output": ..., "code_fix_acknowledgement": ...}.

Adding a New Agent Node

1

Add a prompt file

Create config/prompts/my_agent_prompt.txt. Use {feature_request} and {feedback_section} as placeholders. Call sanitize_for_format(state["feature_request"]) before interpolation.

2

Write the node function

Return only the keys you produce: return {"my_output": output}. Import call_llm, load_prompt, and build_feedback_section from src.agents.base.

3

Add the field to AgentState

Add my_output: MyOutput | None to AgentState in src/state/state.py.

4

Wire it into the graph

Register with builder.add_node() and connect with builder.add_edge() or builder.add_conditional_edges() in src/graph/graph.py.
